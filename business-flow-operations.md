# 北海365分类信息系统 — 运营方后台业务流程

> 本文档从运营人员的视角出发，按日常工作场景梳理后台管理端的完整业务流程。
> 运营人员的工作可以归纳为三个核心职责：**审内容、管用户、配规则**。

# 一、登录与进入后台

## 1.1 后台登录

```
打开后台管理地址 → 输入账号 + 密码 → 提交登录
  → 建立登录态 → 进入后台首页
```

- 后台账号独立于前台用户系统，数据表 `s_admin_manager`
- 登录凭证：`manage_name` + `password`（加 `salt` 加密校验）
- 登录成功/失败均记录日志（`s_admin_manager_log`）
- 当前不涉及手机号登录、权限角色、密码找回

## 1.2 退出登录

```
点击退出登录 → 服务端销毁登录态 → 回到登录页
```

# 二、信息审核（日常核心工作）

> 信息列表是运营人员的主战场，按信息状态分为六个工作页签，每个页签对应不同的工作模式。

## 2.1 工作台总览

路径：`分类 → 分类信息 → 信息列表`

### 状态页签

```
┌────┬────┬────┬────┬─────┬─────┬─────┐
│全部│通过│待审│拒绝│待支付│已失效│已删除│
└────┴────┴────┴────┴─────┴─────┴─────┘
```

| 页签 | 对应状态 | 工作模式 |
|------|---------|---------|
| 全部 | 无过滤 | 全局检索与概览 |
| **待审** | `status=2` | **审核处理**（最高频） |
| **通过** | `status=1` | **上线运营管理** |
| 拒绝 | `status=3` | 恢复或清理 |
| 待支付 | `is_pay=2` | 支付异常处理 |
| 已失效 | `status=4` | 重新激活或清理 |
| 已删除 | `status=5` | 审计查看 |

### 筛选区

所有页签共用基础筛选：

| 筛选项 | 说明 |
|--------|------|
| 关键词 | 标题/内容模糊搜索 |
| 信息 ID | 精确查找 |
| 用户名 | 发布人筛选 |
| 联系电话 | 联系方式筛选 |
| 分类 | 分类树下拉 |
| 来源 | APP / 小程序 / PC / 后台 |
| 发布时间 | 时间范围 |

通过页额外筛选：时间排序、置顶类型、刷新类型、分类首页状态

操作按钮：搜索、导出数据、清空筛选、更多筛选

### 信息卡片展示

列表不是简单表格，而是"信息卡片 + 状态栏 + 操作栏"结构。每条卡片展示：

```
┌───────────────────────────────────────────────────────────┐
│ 【跳蚤市场-二手手机】                                       │
│ 标题：我有一部二手手机                    ID: 181969        │
│ 发布人：南归的雁                                           │
│ 更新时间：2026-03-25 16:02     发布时间：2026-03-25 16:02   │
│ 来源：PC    浏览量：0    联系量：0    [打开小程序]           │
│─────────────────────────────────────────────────────────── │
│ 标题：我有一部二手手机                                      │
│ 地址：北海市海城区                                          │
│ 价格：1000                                                 │
│ 类型：苹果                    ┌──────┐                     │
│ 详细描述：九成新，自用...      │ 缩略图 │                    │
│ 标签：经济实惠、九成新         └──────┘                     │
│ 联系人：张三  手机号：138****                                │
├───────────────────────────────────────────────────────────┤
│ 状态：是否成交：否                                          │
├───────────────────────────────────────────────────────────┤
│ [通过] [拒绝] [编辑] [更多]                                 │  ← 操作区（按页签不同变化）
```

审核人员**无需进入详情即可快速判断内容质量**。

## 2.2 待审页 — 审核处理

> 这是运营人员最高频的工作页面。

### 操作一：审核通过

```
点击"通过" → 弹出审核弹窗
  ├── 在分类首页显示（勾选项）
  ├── 通知作者（勾选项）
  └── [确定]
```

结果：`status` 改为 `1`（通过），记录 `pass_at` 时间

### 操作二：审核拒绝

```
点击"拒绝" → 弹出审核弹窗
  ├── 操作理由（下拉选择预设理由库）
  ├── 自定义理由（文本输入，优先级高于预设理由）
  ├── 退款给作者（显示应退金额，如"免费"表示无退款）
  ├── 退套餐给作者（如"无可退套餐"表示不可退）
  ├── 通知作者（勾选项）
  └── [确定]
```

结果：`status` 改为 `3`（拒绝），`deny_reason` 记录拒绝原因，费用原路退回

**预设理由示例**：
- 类别错误，请选择对应类别重新发布
- 禁止发布出售号码信息，多次发布系统自动封号
- 信息违反广告法，请勿使用"最"等字眼
- 主题不明或描述不清，请重新编辑后发布
- 商品单价虚假，请核实后发布
- 多个账户发布大量相同性质的内容，视为灌水信息

### 操作三：编辑

```
点击"编辑" → 打开信息编辑表单（回填已有内容）
  → 修改内容 → 提交
```

- 待审状态编辑后，状态保持待审

### 操作四：批量审核

```
勾选多条信息（或全选）→ 底部操作栏
  ├── [通过] — 批量通过
  ├── [拒绝] — 批量拒绝
  └── [删除] — 批量删除
```

---

## 2.3 通过页 — 上线运营管理

> 信息通过审核后，工作重心从"审核处理"切换为"运营管理"。

| 操作 | 说明 |
|------|------|
| **拒绝** | 已通过的信息可重新拒绝下架（审核结果可回退） |
| **编辑** | 修改内容；若触发敏感词自动转为待审，否则保持通过 |
| **刷新** | 让信息重新出现在前台列表顶部（对应分类模块刷新能力） |
| **加热** | 通过马甲互动系统提升信息热度（当前不深挖） |
| **转发** | 自动转发信息到其他渠道（当前不深挖） |
| **添加客户至CRM** | 将信息发布者同步到 CRM 线索系统（当前不深挖） |

通过页额外筛选能力：

```
时间排序：发布时间倒序 / 更新时间倒序
更多筛选：
  ├── 置顶类型
  ├── 刷新类型
  └── 分类首页状态
```

**编辑后的状态流转规则**：
- 通过态编辑 → 未触发敏感词 → 状态不变
- 通过态编辑 → 触发敏感词 → 自动进入待审

---

## 2.4 拒绝页 — 恢复或清理

| 操作 | 说明 |
|------|------|
| **通过** | 恢复已拒绝的信息为通过状态（重新上线） |
| **删除** | 逻辑删除（`status=5`） |

拒绝不是终态，运营可以随时恢复。

---

## 2.5 待支付页 — 支付异常处理

| 操作 | 说明 |
|------|------|
| **通过** | 仅改变信息审核状态，**不会更改商户平台的订单状态** |

页面顶部有黄色警示提示：

> 待支付下的信息，通过后仅改变信息状态，不会更改各个商户平台下的订单状态，介意对账有出入的话，[通过] 操作慎用！！！

**关键规则**：信息审核状态与支付订单状态是两套独立状态机。

---

## 2.6 已失效页 — 重新激活或清理

| 操作 | 说明 |
|------|------|
| **刷新** | 重新激活失效信息，恢复上线 |
| **编辑** | 修改内容后继续处理 |
| **删除** | 逻辑删除 |

已失效页额外有"状态"下拉二次筛选（默认全部）。

---

## 2.7 已删除页 — 审计查看

- 仅供查看，当前截图未见操作按钮
- 卡片展示"删除人"信息（如 `删除人：用户`），区分用户自删和管理员删除
- 用于审计追溯

# 三、评论审核

> 评论审核有两个入口，日常可选择更顺手的方式。

路径：`分类 → 评论列表`

## 3.1 入口一：评论列表统一审核

### 页面结构

```
状态页签：[待审] [通过] [不通过]

筛选区：
  ├── 用户筛选类型（用户名 / 用户ID 等）
  ├── 用户输入框
  ├── 信息 ID
  ├── 评论关键词
  └── [搜索]

列表字段：
  ┌──────┬──────────────┬──────┬──────────────┬──────┬──────────┐
  │ 选择 │   评论内容    │ 时间 │ 分类信息标题  │ 用户 │   操作   │
  └──────┴──────────────┴──────┴──────────────┴──────┴──────────┘
```

### 单条操作

| 操作 | 说明 |
|------|------|
| **通过** | 评论审核通过，前台公开展示 |
| **不通过** | 评论审核拒绝，前台不展示 |
| **详情** | 打开评论所属信息的详情弹窗（不是评论详情） |

- 部分评论文字可能被标红（疑为敏感词高亮）
- 评论状态：待审 → 通过 / 不通过

## 3.2 入口二：信息详情弹窗内联审核

```
信息列表 → 任意信息卡片 → 点击"详情"
  → 信息详情弹窗（底部有评论区）
    → 该信息下的所有评论列表
      → 每条评论可直接 [通过] / [不通过]
```

**使用场景**：审核信息时顺便处理该信息下的评论，无需切换到评论列表页。

### 信息详情弹窗内容

```
┌────────────────────────────────────────┐
│  信息详情弹窗                           │
├────────────────────────────────────────┤
│  图片                                  │

│  更新时间 / 浏览量                      │
│  地址 / 结构化字段                      │
│  详细描述                              │
├────────────────────────────────────────┤
│  评论区                                │
│  ┌─ 用户A：评论内容    [待审]  [通过] [不通过] │
│  ├─ 用户B：评论内容    [通过]                 │
│  └─ 用户C：评论内容    [待审]  [通过] [不通过] │

```

# 四、代发布信息

> 运营人员可以在后台代替某个前台用户发布信息。

路径：`分类 → 分类信息 → 发布信息`

## 4.1 发布流程（三步）

```
第1步：选择分类
  → 点击一级分类（如"跳蚤市场"）
    → 弹出二级分类弹窗
      → 选择二级分类（如"二手手机"）

第2步：选择发布人 + 发布身份
  → 弹窗1：搜索并选择一个前台用户（输入用户名）
    → 弹窗2：选择发布身份
      ├── 个人
      └── 企业认证
      （若用户组不满足，提示需先将用户加入对应用户组）

第3步：填写信息
  → 动态表单（根据分类模板生成字段）
    → 填写内容 + 上传图片
      → 提交发布
```

### 后台专属字段

与前台发布相比，后台额外支持：

| 字段 | 说明 |
|------|------|
| 发布到期时间 | 后台可直接设置到期时间 |
| 背景色 | 信息展示背景色 |
| 视频号 | 启用/禁用 |
| 置顶推广 | 后台免费置顶（无需付费） |

### 关键规则

- 后台发布是**代发**，不是管理员自己发。必须绑定一个前台用户
- 发布身份依赖用户组体系（`s_group` + `s_group_user`）
- 表单是动态生成的，不同分类、不同模板字段不同

# 五、审核理由库管理

> 维护审核拒绝时的预设理由模板。

路径：`分类 → 分类信息 → 审核操作理由`

## 5.1 页面结构

```
┌────┬───────────────────────────────────────────┬──────────┬──────┐
│排序│                 文案                       │ 创建时间  │ 操作 │
├────┼───────────────────────────────────────────┼──────────┼──────┤
│ 1  │ 类别错误，请选择对应类别重新发布            │ 2026-01  │ 编辑 │
│ 2  │ 禁止发布出售号码信息，多次发布自动封号       │ 2026-01  │ 删除 │
│ 3  │ 信息违反广告法，请勿使用"最"等字眼          │ 2026-01  │      │
│ ...│                                           │          │      │
└────┴───────────────────────────────────────────┴──────────┴──────┘

页面顶部：[新增理由]
排序方式：长按可拖动排序
```

## 5.2 操作流程

| 操作 | 流程 |
|------|------|
| **新增理由** | 点击"新增" → 弹窗输入理由文案 → 确定 |
| **编辑理由** | 点击"编辑" → 弹窗修改文案 → 确定 |
| **删除理由** | 点击"删除" → 逻辑删除（`status=0`） |
| **调整排序** | 长按拖动行，调整显示顺序 |

## 5.3 与审核弹窗的关系

```
审核理由库（本页维护）
      │
      ▼
审核拒绝弹窗 → "操作理由"下拉 → 从理由库读取预设选项
                "自定义理由"输入 → 可覆盖预设理由
                      │
                      ▼
              最终写入 s_fenlei_info.deny_reason
```

- 理由库是配置源，审核结果是最终落点
- 预设理由和自定义理由同时存在时，自定义理由优先
- 数据表：`s_fenlei_audit_reason`（id, content, sort, status）

# 六、分类管理

> 分类是整个平台的骨架，决定了前台展示什么内容、发布用什么表单、收什么费用。

路径：`分类 → 分类信息 → 分类列表` / `全局设置`

## 6.1 分类树管理

### 分类列表页

- 维护一级/二级分类的树结构
- 查看分类启用状态、排序
- 操作入口：基础设置、字段设置、费用及权限

### 分类层级规则

```
一级分类（容器，如"跳蚤市场"）
  ├── 二级分类（叶子，如"二手手机"）← 发布实际落点
  ├── 二级分类（叶子，如"家用电器"）
  └── ...
```

- 一级分类已有二级分类后，只保留"编辑分类"和"添加二级分类"
- 完整业务配置落在叶子分类上

## 6.2 单分类基础设置

| 配置项 | 说明 |
|--------|------|
| 分类名称 | 前台显示名 |
| 分类图标 | 图片上传（七牛云） |
| 排序 | 前台展示顺序 |
| 标签色 | 分类标签颜色 |
| 发布须知 | 用户发布前展示的提示文案 |
| 模板选择 | 决定该分类的系统字段组合 |

### 模板列表

| 模板 ID | 模板名称 | 典型系统字段 |
|---------|---------|------------|
| 1 | 图文分享 | pictures, detail, tags, linkman, mobile |
| 2 | 电话突出 | title, address, pictures, detail, tags, linkman, mobile |
| 3 | 价格突出 | title, address, pictures, price, detail, tags, linkman, mobile |
| 4 | 简易招聘 | position, company, address, paysection, paytype, intro, tags, linkman, mobile |
| 5 | 拼车模式 | source, dest, time, timemark, detail, tags, linkman, mobile |
| 6 | 经典单图 | title, pictures, detail, tags, linkman, mobile |
| 7 | 垂直招聘 | position, positiontype, company, region, address, paysection, paytype, intro, tags, linkman, mobile |
| 8 | 简易零工 | 与简易招聘共用前缀 |

## 6.3 单分类字段设置

### 字段组成

```
发布/编辑表单
  ├── 系统字段（由模板决定，不可增删类型）
  └── 自定义字段（运营可添加）
```

| **删除账号** | 移除后台账号 |

## 12.3 操作日志

```
操作日志列表（s_admin_manager_log）
  → 操作人（manage_id）
  → 操作类型（type）+ 操作动作（action）
  → 操作对象（pk）
  → 操作 IP、UA
  → 操作时间
```

- 记录所有后台操作：登录、审核、编辑、删除等
- 支持按管理员、操作类型、时间范围筛选
- 用于安全审计和操作追溯

# 十二、权限管理

> 控制后台管理员的菜单访问和操作权限。

路径：`权限管理`

## 13.1 权限体系结构

```
角色（s_project_group_role）
  ├── 角色名称
  ├── 权限集合（JSON 格式，存储可访问的菜单和操作）
  └── 关联管理员
        │
        ▼
管理员账号（s_admin_manager）
  └── access 字段：权限标识字符串
```

## 13.2 角色管理

| 操作 | 说明 |
|------|------|
| **新增角色** | 设置角色名称，勾选可访问的菜单和操作权限 |
| **编辑角色** | 调整角色的权限范围 |
| **删除角色** | 移除角色（需先解绑关联管理员） |

### 权限粒度

| 层级 | 示例 |
|------|------|
| 菜单权限 | 是否可见"信息列表"、"套餐设置"、"用户管理"等菜单 |
| 操作权限 | 是否可执行"审核通过"、"审核拒绝"、"编辑信息"、"删除"等操作 |
| 数据权限 | 是否可查看所有分类还是仅特定分类的数据 |

## 13.3 管理员分配角色

```
后台用户管理 → 选择管理员 → 分配角色
  → 管理员获得角色对应的权限集合
  → 登录后只能看到和操作有权限的菜单和功能
```

## 13.4 典型角色配置示例

| 角色 | 菜单权限 | 操作权限 |
|------|---------|---------|
| 超级管理员 | 全部菜单 | 全部操作 |
| 内容审核员 | 信息列表、评论列表 | 审核通过/拒绝、编辑 |
| 商家运营 | 商家管理 | 商家审核、添加商家、商家信息维护 |
| 财务人员 | 交易管理、套餐设置 | 查看流水、处理提现 |
| 客服人员 | 用户管理、信息列表 | 查看用户、封禁、编辑信息 |

# 十四、关键业务规则速查

| # | 规则 | 说明 |
|---|------|------|
| 1 | 审核不是单纯改状态 | 通过可设分类首页展示，拒绝可退费/退套餐/通知作者 |
| 2 | 编辑触发敏感词会重审 | 通过态信息编辑后触发敏感词，自动回到待审 |
| 3 | 审核状态与支付状态独立 | `status` 和 `is_pay` 是两套状态机，互不干扰 |
| 4 | 拒绝不是终态 | 被拒绝的信息可随时恢复为通过 |
| 5 | 代发布是"帮用户发" | 后台发布必须绑定一个前台用户和发布身份 |
| 6 | 套餐上线需两步 | 开启"前端显示" + 点击"套餐发布"，缺一不可 |
| 7 | 理由库是配置源 | 审核拒绝的最终理由写入 `deny_reason`，预设理由只是模板 |
| 8 | 全局配置可被覆盖 | 单分类配置有值时覆盖全局配置 |
| 9 | 评论审核有两个入口 | 评论列表统一审核 + 信息详情弹窗内联审核 |
| 10 | 所有操作留痕 | 审核、编辑、删除、封禁等操作全部写入操作日志 |
| 11 | 商家与用户一对一 | 一个前台用户只能对应一个商家（`s_stores.user_id` 唯一索引） |
| 12 | 后台添加商家跳过审核 | 后台直接创建的商家无需经过认证审核流程 |
| 13 | 权限按角色分配 | 管理员通过角色获得权限，角色以 JSON 存储权限集合 |
